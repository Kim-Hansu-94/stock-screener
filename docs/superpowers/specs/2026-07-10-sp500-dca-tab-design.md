# S&P 500 적립 탭 설계 (v2 — 재설계)

- 날짜: 2026-07-10 (v1 조정 매수 설계를 백테스트 검증 후 전면 재설계)
- 상태: 설계 승인됨 (구현 전)
- 대상: 김한수의 보물지도에 S&P 500 정기 적립 탭(`/sp500`) 추가

## v1에서 바뀐 이유

v1은 "조정 단계(-5/-10/-20/-30%)마다 예비자금을 분할 투입"하는 설계였다.
구현 전 백테스트(`pipeline/research/backtest_sp500_stages.py`, 2000~2026 실데이터)
결과 **정기 적립이 조정 매수보다 일관되게 우위**였다:

| 전략 | 2000년~ | 2010년~ |
|---|---|---|
| A. 정기 적립 (매월 무조건 매수) | 4.469배 | 3.228배 |
| B. 단계별 분할 (v1 설계) | 4.099배 | 2.829배 |
| C. 단일 임계값 (-10% 일괄) | 4.001배 | 2.965배 |

원인: 조정 기준인 고점이 계속 올라가므로 "고점 대비 할인"이 "기다리기 전
가격보다 비싼 매수"가 되는 경우가 많고, 대기 현금의 기회비용이 복리로 쌓인다.
사용자의 자금 성격이 "매월 생기는 저축"이므로 데이터의 답은 조건 없는 매월
매수다. 이에 따라 탭의 성격을 **타이밍 판단 → 실행 추적**으로 전환한다.
상세 수치·해석은 `docs/backtest-guide.md` 참고.

## 확정된 결정

| 항목 | 결정 |
|---|---|
| 전략 | TIGER 미국S&P500(360750)을 매월 조건 없이 매수 |
| 탭 핵심 | 매수 기록 입력 + 적립 현황·수익률 추적 (Supabase 저장, 기기 무관) |
| 조정 정보 | 참고 표시만 — "52주 고점 대비 -X%" 한 줄 + 지수 차트. 행동 조건 아님 |
| 폐기 | v1의 조정 단계·사이클·시그널 이력·매수 플랜·예비자금 계산 전부 |
| 아키텍처 | 파이프라인이 매일 데이터 수집 → Supabase → 프론트 표시. 매수 기록만 프론트 서버 액션으로 쓰기 |

## 1. 탭 화면 구성 (`/sp500`)

내비게이션 라벨: `S&P 500 적립`. 위에서 아래:

1. **이번 달 매수 체크 히어로** — 이번 달(KST) 매수 기록이 있으면
   "✅ N월 적립 완료", 없으면 "이번 달 아직 매수 전입니다" + TIGER 현재가.
2. **매수 기록 입력** — 날짜(기본 오늘)·수량·단가 입력 → Supabase 저장.
   잘못 넣은 기록은 행에서 삭제 가능.
3. **적립 현황** — 총 투자금, 보유 수량, 평단가, 평가액(최신 종가 기준),
   수익률 + 전체 매수 이력 표.
4. **가격 차트 + 매수 마커** — TIGER 종가 라인 차트(약 3년)에 매수 시점을
   마커로 표시. 꾸준한 실행이 시각적으로 보이는 동기부여 장치.
5. **시장 현황 (참고)** — "S&P 500, 52주 고점 대비 -X.X%" + 지수 라인 차트.
   조정 중이어도 "계획대로 매수하는 달입니다" 고정 문구 — 정보이지 조건이 아님.
6. **왜 조건 없이 사는가** — 위 백테스트 표 상시 표시. 하락장에 흔들릴 때
   보는 근거 섹션.

## 2. 데이터

### Supabase 테이블

```
sp500_daily        -- 지수 참고 데이터 (시장 현황 섹션)
  date PK, close, high_52w, drawdown_pct

sp500_purchases    -- 매수 기록 (프론트 서버 액션이 쓰기)
  id PK(identity), date, shares(>0), price(>0), created_at

sp500_etf_quotes   -- ETF 시세 (최신만 유지)
  ticker PK, name, close, change_pct, currency, as_of, updated_at
```

TIGER 360750 일봉 히스토리는 기존 `stock_price_history`(market='KR') 재활용.
v1의 `sp500_signals` 테이블과 `sp500_daily`의 stage/cycle_id/보조지표 컬럼은
만들지 않는다.

### 파이프라인

`pipeline/src/sp500_monitor.py`를 일일 파이프라인 끝에 한 단계로 추가:

- ^GSPC 종가 약 3년치 수신(yfinance) → 52주 고점·하락률 계산 → `sp500_daily`
  upsert (최초 전체, 이후 최근 30일 창만).
- TIGER 360750 일봉 3년치(FinanceDataReader) → `stock_price_history` 저장,
  최신 종가·등락률 → `sp500_etf_quotes` upsert.
- 실패 시 해당 단계만 건너뛰고 파이프라인 전체는 계속. 프론트는 기준일 표시.

### 프론트 쓰기 경로 (매수 기록)

Next.js 서버 액션으로 `sp500_purchases` insert/delete. 검증: shares>0,
price>0, date 유효. 별도 캐시 태그(`sp500-purchases`)를 두어 기록 변경 시
그 태그만 무효화. 나머지 조회는 기존 `SCREENER_CACHE_TAG` + 파이프라인
revalidate 흐름 그대로.

## 3. 계산 (프론트 순수 함수)

- 총 투자금 = Σ(shares × price), 보유 수량 = Σ shares, 평단가 = 투자금/수량
- 평가액 = 보유 수량 × TIGER 최신 종가, 수익률 = 평가액/투자금 − 1
- 이번 달 매수 여부 = KST 기준 당월 기록 존재
- vitest로 커버

## 4. 에러 처리 & 테스트

- 파이프라인: yfinance/FDR 실패 시 해당 항목만 건너뜀 (v1과 동일 원칙).
- 서버 액션: 검증 실패 시 저장하지 않고 에러 메시지 반환.
- pytest: 하락률 계산, 행 직렬화. vitest: 적립 통계, 당월 매수 판정.

## 범위 밖 (YAGNI)

- 매수 알림/푸시, 자동 매수 연동
- 여러 종목 지원 (360750 단일 종목 상수)
- 배당·분배금 추적, 세금 계산
- 조정 단계별 행동 가이드 (v1에서 폐기)
